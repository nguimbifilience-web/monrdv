## 1. Etat initial du projet

## 2. Problemes identifies et corriges

### 2.1 BUG CRITIQUE - ClassNotFoundException

| Fichier | Probleme | Correction |
|---------|----------|------------|
| `app/Http/Controllers/DashboardController.php` | Import `App\Models\Rendezvous` (minuscule) | Corrige en `App\Models\RendezVous` (PascalCase) |
| Meme fichier, ligne 15 | Appel `Rendezvous::with(...)` | Corrige en `RendezVous::with(...)` |

**Impact** : Le dashboard crashait systematiquement avec une erreur fatale PHP.

---

### 2.2 ROUTES MANQUANTES - Erreurs 404

| Route | Controleur | Action |
|-------|-----------|--------|
| `patients.*` | PatientController | Ajout `Route::resource('patients', ...)` |
| `specialites.*` | SpecialiteController | Ajout `Route::resource('specialites', ...)->except(...)` |
| `assurances.*` | AssuranceController | Ajout `Route::resource('assurances', ...)->except(...)` |

**Fichier modifie** : `routes/web.php`
**Impact** : La sidebar contenait des liens vers ces routes, mais elles n'existaient pas. Cliquer dessus generait une erreur 404.

### 2.3 LAYOUT INCOMPATIBLE - Pages blanches

11 vues Blade utilisaient `@extends('layouts.app')` avec `@section('content')`.
Or, `layouts/app.blade.php` utilise `{{ $slot }}` (composant Blade Breeze), ce qui est **incompatible** avec `@section`.
Le layout correct est `layouts/master.blade.php` qui utilise `@yield('content')`.

| `patients/create.blade.php` | `layouts.app` | `layouts.master` |
| `patients/show.blade.php` | `layouts.app` | `layouts.master` |

| `rendezvous/create.blade.php` | `layouts.app` | `layouts.master` |

### 2.4 ASSURANCES EN DUR - Donnees statiques au lieu de la BDD

**Fichier** : `app/Http/Controllers/PatientController.php`

Les methodes `create()` et `edit()` contenaient des assurances codees en dur :

```php
// AVANT (incorrect)
$assurances = collect([
    (object)['id' => 1, 'nom' => 'CNAMGS'],
    (object)['id' => 2, 'nom' => 'AXA GABON'],
    (object)['id' => 3, 'nom' => 'NSIA'],
    (object)['id' => 4, 'nom' => 'ASCOMA']
]);

// APRES (correct)
$assurances = Assurance::all();
```

**Impact** : Les assurances ajoutees via l'interface d'administration n'apparaissaient jamais dans le formulaire patient.

### 2.6 FILTRES PATIENTS NON FONCTIONNELS

**Fichier** : `app/Http/Controllers/PatientController.php`

```php
// AVANT - filtres ignores
public function index() {
    $patients = Patient::with('medecin')->latest()->paginate(10);
}

// APRES - filtres actifs
public function index(Request $request) {
    $patients = Patient::with('medecin')
        ->filter($request->only(['search', 'medecin_id', 'est_assure']))
        ->latest()
        ->paginate(10)
        ->withQueryString();
}
```

**Impact** : La recherche et les filtres de la vue `patients/index` ne produisaient aucun resultat filtre. `withQueryString()` ajoute pour conserver les filtres lors de la pagination.

## 4. Points d'attention restants

| # | Sujet | Severite | Detail |
|---|-------|----------|--------|
| 1 | Migration dupliquee `est_assure` | Faible | Deux migrations ajoutent la meme colonne. La seconde a un garde `hasColumn` donc pas de crash, mais c'est du bruit. |
| 2 | Migration mal nommee | Faible | `2026_03_25_124511_add_est_assure_to_patients_table.php` ajoute en realite `description` a `assurances`, pas `est_assure` a `patients`. |
| 3 | Migrations admin vides | Faible | `add_is_admin_to_users_table` et `add_admin_fields_to_users_table` sont vides (up/down ne font rien). |
| 4 | PHP absent du PATH | Info | `php artisan` n'est pas executable depuis le terminal actuel. Les routes n'ont pas pu etre validees par artisan. |
| 5 | CSRF sur toggle AJAX | Moyen | `toggleDispo` recoit des requetes POST via axios. S'assurer que le meta tag CSRF est bien lu par axios globalement. |

## 19. Securisation du deploiement - Niveau 1 (10 avril 2026)

Demande utilisateur : "mettre en place un systeme de securite pour proteger l'application
en ligne". Contexte : environnement de TEST (pas de vraies donnees patient), donc niveau
de securite proportionne (pas de 2FA ni packages composer). Aucune modification du code
applicatif - tout passe par les fichiers `.htaccess`.

### 19.1 Mesures ajoutees dans `public/.htaccess`

**Headers HTTP de securite (mod_headers)** :
- `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload` (HSTS 1 an)
- `X-Frame-Options: DENY` (anti-clickjacking)
- `X-Content-Type-Options: nosniff` (anti-MIME sniffing)
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: geolocation=(), microphone=(), camera=()...` (desactive APIs sensibles)
- `Cross-Origin-Opener-Policy: same-origin`
- `Cross-Origin-Resource-Policy: same-site`
- `X-Powered-By` et `Server` masques (`Header unset`)

**Blocage des fichiers sensibles** (`<FilesMatch>`) :
- `.env*`, `.git*`, `composer.json/lock`, `package*.json`, `webpack/vite/tailwind/postcss.config.js`
- `phpunit.xml*`, `artisan`, `server.php`, `.htpasswd`, `.user.ini`
- `README*`, `CHANGELOG*`, `LICENSE*`
- Extensions : `*.log`, `*.bak`, `*.backup`, `*.old`, `*.orig`, `*.swp`, `*.tmp`, `*.sql`,
  `*.sqlite*`, `*.yml`, `*.yaml`, `*.ini`, `*.conf`, `*.config`, `*.lock`, `*.md`

**Blocage des dossiers caches** (`RedirectMatch 404`) :
- `/.git`, `/.svn`, `/.hg`, `/.bzr`

**Blocage des methodes HTTP inutilisees** :
- `TRACE`, `TRACK`, `CONNECT` (anti cross-site tracing)

**Blocage des User-Agents de scanners malveillants** (regex case-insensitive) :
- nikto, sqlmap, masscan, nmap, fimap, nessus, whatweb, openvas, jbrofuzz, libwhisker,
  webshag, w3af, arachni, x-scanner, scanner, skipfish, ZmEu, Morfeus, zgrab, gobuster,
  dirbuster, wfuzz, hydra, Wprecon
- User-Agent vide aussi bloque

**Hardening serveur** :
- `Options -Indexes -MultiViews` (pas d'enumeration de dossiers)
- `php_flag expose_php Off` (masque la version PHP dans les headers)

### 19.2 Mesures ajoutees dans `deploy/htaccess-laravel-deny` (renforce 2026-04-10)

Fichier a placer dans `htdocs/laravel/.htaccess` (deja en place sur le serveur, mais
nouvelle version a re-uploader pour beneficier des protections supplementaires).
Strategie : defense en profondeur sur 8 niveaux. Si l'un tombe (config Apache, override,
faille), les autres tiennent.

| Niveau | Mesure | Protege contre |
|--------|--------|----------------|
| 1 | `Require all denied` (Apache 2.4+) | Acces HTTP direct a tout fichier |
| 2 | `Order deny,allow` + `Deny from all` (Apache 2.2 fallback) | Idem sur vieux Apache |
| 3 | `FilesMatch` bloquant `.php/.phtml/.phar/.php3-8/.pht/.phtm/.cgi/.pl/.py/.jsp/.asp/.aspx/.sh/.exe/.bat/.cmd` (case-insensitive) | Extensions executables meme si Niveau 1/2 tombe |
| 4 | `SetHandler None` + `ForceType text/plain` sur les memes extensions | Apache ne sait plus quoi executer, sert en texte brut |
| 5 | `RemoveHandler` / `RemoveType` / `AddType text/plain` via `mod_mime` | Desassociation complete du handler PHP/CGI |
| 6 | `php_admin_flag engine off` + `allow_url_fopen off` + `allow_url_include off` (mod_php/7/8) | Moteur PHP desactive ; `php_admin_flag` NON surchargeable par `ini_set()` contrairement a `php_flag` |
| 7 | `Options -Indexes -ExecCGI -MultiViews -Includes -FollowSymLinks` | Pas de listing, pas de CGI, pas de negociation, pas de SSI, pas de traversal symlink |
| 8 | `X-Content-Type-Options: nosniff` + `X-Frame-Options: DENY` + unset `X-Powered-By` | Fuite de type MIME, clickjacking, fingerprinting |

Changements cles par rapport a la version precedente :
- Extensions bloquees etendues (+ cgi/pl/py/jsp/asp/aspx/sh/exe/bat/cmd, + case-insensitive)
- Ajout de `SetHandler None` et `ForceType text/plain` (Niveau 4, nouveau)
- Ajout de `RemoveHandler`/`RemoveType`/`AddType` via `mod_mime` (Niveau 5, nouveau)
- `php_flag engine off` remplace par `php_admin_flag engine off` (non-surchargeable)
- Ajout `allow_url_fopen off` et `allow_url_include off` (anti-RFI)
- `Options` etendu avec `-Includes` (anti-SSI) et `-FollowSymLinks` (anti-traversal)
- Ajout de headers defensifs (Niveau 8, nouveau)

## 16. Script chmod 775 jetable pour deploiement mutualise (10 avril 2026)

Demande utilisateur : appliquer chmod 775 sur les dossiers Laravel inscriptibles par
le serveur web, en sachant que :
- Windows (NTFS) ne stocke pas les bits POSIX rwx, donc impossible en local
- L'hebergement page.gd est mutualise sans acces SSH
- Le workflow actuel utilise FTP pour uploader

Solution retenue (option A standalone) : nouveau script `deploy/chmod.php`.

| Element | Valeur |
|---------|--------|
| Fichier cree | `deploy/chmod.php` |
| Emplacement serveur | `htdocs/chmod.php` (a cote de `migrate.php`) |
| Cibles chmod | `htdocs/laravel/bootstrap/cache` et `htdocs/laravel/storage` (recursif) |
| Mode applique | `0775` |
| Token securite | genere via `openssl rand -hex 32` le 2026-04-10 |
| URL d'appel | `https://mnrdv.page.gd/chmod.php?token=<token>` |
| Cycle de vie | jetable - a SUPPRIMER du serveur immediatement apres execution |

Implementation technique :
- Token compare en strict (`===`), 403 sinon
- Parcours `RecursiveIteratorIterator` avec `SELF_FIRST` pour traiter le dossier racine
  puis ses descendants (dossiers + fichiers)
- `@chmod` silencieux avec compteur ok/ko et listing des echecs
- Sortie texte brut, rappel final de suppression du fichier

Procedure utilisateur :
1. Uploader `deploy/chmod.php` vers `htdocs/chmod.php` via FileZilla
2. Visiter l'URL avec le token (1 fois par deploiement si les droits sont reecrases)

## 24. Audit securite 2026-04-20 — Correctif #3 : authorize() ordonnances/examens

**Contexte** : l'audit avait signale que `OrdonnanceController::authorizeView` et
`FeuilleExamenController::authorizeView` verifiaient uniquement `medecin_id` /
`patient_id`, sans `clinic_id`.

**Analyse approfondie** :
- `Ordonnance` et `FeuilleExamen` ont le trait `BelongsToClinic` + `ClinicScope`,
  donc le route model binding filtre deja par `clinic_id` → l'IDOR direct n'est
  pas exploitable en l'etat.
- **Mais** les regles de validation `exists:patients,id` et
  `exists:consultations,id` utilisees dans `store()` font des queries directes
  (pas de scope applique). Un medecin de clinique A pouvait theoriquement
  creer une ordonnance dont `patient_id` pointait vers un patient de clinique B
  → ordonnance orpheline incoherente (clinic_id assigne automatiquement par
  `BelongsToClinic::creating`, mais patient_id hors clinique).

**Correctifs appliques** (local) :
1. `app/Http/Controllers/OrdonnanceController.php` :
   - Import `Illuminate\Validation\Rule`
   - `store()` : `exists:patients,id` → `Rule::exists('patients','id')->where('clinic_id', $clinicId)`
   - `store()` : idem pour `consultation_id`
   - `authorizeView()` : ajout check `$ordonnance->clinic_id === $user->clinic_id` en defense en profondeur
2. `app/Http/Controllers/FeuilleExamenController.php` : meme refactor identique.

**Notes** :
- Le check `clinic_id` est conditionne par `$user->clinic_id` non-null : les
  super_admin (sans clinic_id) ne sont pas bloques, mais ils ne peuvent de toute
  facon pas acceder aux routes `espace-medecin/` ou `espace-patient/` (middleware
  `role:medecin|patient`).
- Defense en profondeur : si un jour quelqu'un ajoute `withoutGlobalScopes()`,
  le check reste actif.

**Non traite dans ce patch** (meme pattern a auditer plus tard) :
- `ConsultationController::getPatientInfo($id)` et `creneauxDisponibles` : le
  scope protege deja, mais ajouter un check explicite clinic_id serait coherent.
- Tous les `Rule::exists(...)` du projet devraient etre audites pour ajouter
  `where('clinic_id', ...)` sur les ressources multi-tenant.

**Etat** : correctifs #1, #2, #3 faits en local. Restent #4 MDP patients
predictibles, #5 logs donnees medicales.

## 25. Audit securite 2026-04-20 — Correctif #4 : MDP predictibles

**Probleme** :
1. `PatientController::store` generait le MDP patient via
   `strtolower(substr($nom, 0, 3)) . rand(1000, 9999)` → format public
   (3 lettres du nom du patient, publiques) + 4 chiffres seulement
   (10 000 combinaisons → brute force trivial en quelques secondes).
   De plus `rand()` n'est pas cryptographiquement sur.
2. `CompteController`, `MedecinController`, `ClinicController` (x3),
   `SuperAdmin/UserController` (x2) utilisaient `Str::random(8)` :
   8 caracteres alphanumeriques sans symboles. Pas predictible mais
   entropie faible (62^8 ≈ 2.2×10^14) pour un MDP de compte admin medical.

**Correctifs appliques** (local) :
Remplacement dans 5 controllers (9 occurrences) :

| Fichier | Ligne avant | Apres |
|---------|-------------|-------|
| `PatientController.php:56` | `strtolower(substr(...)).rand(...)` | `Str::password(12)` |
| `CompteController.php:23,113` | `Str::random(8)` | `Str::password(12)` |
| `ClinicController.php:119,171,232` | `Str::random(8)` | `Str::password(12)` |
| `MedecinController.php:51` | `Str::random(8)` | `Str::password(12)` |
| `SuperAdmin/UserController.php:52,92` | `Str::random(8)` | `Str::password(12)` |

Ajout de `use Illuminate\Support\Str;` dans `PatientController.php` (manquait).

`Str::password(12)` produit des MDP de 12 caracteres avec
lettres (maj/min) + chiffres + symboles, genere via `random_bytes()`
(cryptographiquement sur). Exemple : `K8n@q2mR#zX!`. Entropie ~72 bits.

**Risque residuel NON traite dans ce patch** :
- Les MDP generes sont toujours **affiches en clair** dans les flash
  messages (`"Identifiants : email@x.fr / K8n@q2mR#zX!"`), car le
  workflow repose sur l'admin transmettant au user. Visible dans :
  cache navigateur, historique, captures ecran, logs serveur si URL
  loggee.
- Solution propre : email SMTP fonctionnel (prod a `MAIL_MAILER=log`
  actuellement, voir audit #2.4), ou flag `must_change_password`
  obligatoire au premier login, ou affichage unique via modal sans
  persistance dans la reponse HTML renvoyee.
- A traiter apres le correctif #5 (logs donnees medicales).

**Etat** : correctifs #1, #2, #3, #4 faits en local. Reste #5 logs
donnees medicales.

## 26. Audit securite 2026-04-20 — Correctif #5 : logs donnees medicales

**Probleme** :
Le systeme d'audit `ActivityLog` stockait en clair :
1. **Descriptions** avec noms/prenoms des patients et medecins :
   - "Patient cree : Jean Dupont"
   - "RDV cree : Jean Dupont avec Dr. Smith le 2026-05-12"
   - "Notes medicales mises a jour pour Jean Dupont" (fait apparaitre un
     nom associe a une donnee medicale implicite)
2. **old_values / new_values (JSON)** : snapshots complets des modeles
   incluant `notes_medicales`, `observations`, `assurance_id`,
   `telephone`, tarifs medecin, etc.

Impact RGPD : en cas de compromission de la BDD, les logs exposaient
  autant de donnees sensibles que les tables metier, voire plus (historique
  des modifications de dossier medical). Les logs, sauvegardes separement,
  devenaient une cible equivalente.

**Correctifs appliques** (local) :

1. **Descriptions anonymisees** (noms -> IDs uniquement) dans 4 controllers :
   - `PatientController.php` (4 appels) :
     - `"Patient cree : Jean Dupont"` -> `"Patient #42 cree"`
     - `"Patient modifie : ..."` -> `"Patient #42 modifie"`
     - `"Patient supprime : ..."` -> `"Patient #42 supprime"`
     - `"Notes medicales mises a jour pour Jean Dupont"` -> `"Notes medicales mises a jour pour patient #42"`
   - `RendezVousController.php` (3 appels) :
     - `"RDV cree : Jean Dupont avec Dr. Smith le 2026-05-12"` -> `"RDV #17 cree : patient #42, medecin #5, date 2026-05-12"`
     - `"RDV #17 modifie"` (inchange)
     - `"RDV #17 confirme pour Jean Dupont"` -> `"RDV #17 confirme"`
   - `MedecinController.php` (3 appels) :
     - `"Medecin cree : Dr. John Smith"` -> `"Medecin #5 cree"`
     - idem modification et suppression
   - `ConsultationController.php` (1 appel) :
     - `"Consultation enregistree : Jean Dupont avec Dr. Smith - 15000 F"` -> `"Consultation #8 creee : patient #42, medecin #5, montant patient 15000 F"`

2. **Suppression des old_values / new_values** pour les modeles contenant
   des donnees medicales ou sensibles :
   - `PatientController::update` : plus de `toArray()` + `request->all()`
   - `RendezVousController::update` : plus de `$oldValues` + `$validated`
   - `MedecinController::update` : plus de `$oldValues` + `$validated`

   Le fait que la modification a eu lieu est toujours trace (user_id, action,
   model_id, ip_address, created_at). Le detail des changements n'est plus
   persiste en clair → defense en profondeur si BDD compromise.

3. **Logs CONSERVES avec email en clair (legitimes)** :
   - `AuthenticatedSessionController` (connexion / deconnexion) : email = identifiant
     d'authentification, necessaire pour l'audit de securite.
   - `CompteController` (creation/modif/reset/suppression compte) : email =
     identite du compte manipule (pas une donnee personnelle tierce, c'est le
     sujet de l'action administrative).
   - `ClinicController` (super admin) : email + nom de clinique (pas PII patient).

**Note** : les anciens logs deja ecrits en BDD contenant des noms/old_values
restent presents (pas de purge). Si besoin d'hygiene complete :
   UPDATE activity_logs SET description = CONCAT('[purge] ', action), old_values = NULL, new_values = NULL
   WHERE model_type IN ('Patient', 'RendezVous', 'Medecin', 'Consultation')
     AND created_at < '2026-04-20';

**Risque residuel** (non traite, pour plus tard) :
- Pas de chiffrement at-rest des descriptions. Les IDs sont reconstituables si
  l'attaquant a aussi les tables metier, mais il y a alors deja fuite directe.
  Pour une protection plus forte : encryption cast sur `description`
  (Laravel `encrypted` cast) ou migration vers une colonne chiffree dediee.
- Pas d'immutabilite des logs. Un attaquant avec acces BDD peut les modifier.
  Solution : append-only log externe (ex: syslog, fichier write-only).

**Etat** : correctifs #1 a #5 du top 5 de l'audit faits en local. Les 45
autres points de l'audit (majeurs + mineurs) restent a traiter au fil
des sessions futures.
