## 1. Etat initial du projet

## 2. Problemes identifies et corriges

### 2.1 BUG CRITIQUE - ClassNotFoundException

| Fichier | Probleme | Correction |
|---------|----------|------------|
| `app/Http/Controllers/DashboardController.php` | Import `App\Models\Rendezvous` (minuscule) | Corrige en `App\Models\RendezVous` (PascalCase) |
| Meme fichier, ligne 15 | Appel `Rendezvous::with(...)` | Corrige en `RendezVous::with(...)` |

**Impact** : Le dashboard crashait systematiquement avec une erreur fatale PHP.

---

### 2.2 ROUTES MANQUANTES - Erreurs 404

| Route | Controleur | Action |
|-------|-----------|--------|
| `patients.*` | PatientController | Ajout `Route::resource('patients', ...)` |
| `specialites.*` | SpecialiteController | Ajout `Route::resource('specialites', ...)->except(...)` |
| `assurances.*` | AssuranceController | Ajout `Route::resource('assurances', ...)->except(...)` |

**Fichier modifie** : `routes/web.php`
**Impact** : La sidebar contenait des liens vers ces routes, mais elles n'existaient pas. Cliquer dessus generait une erreur 404.

### 2.3 LAYOUT INCOMPATIBLE - Pages blanches

11 vues Blade utilisaient `@extends('layouts.app')` avec `@section('content')`.
Or, `layouts/app.blade.php` utilise `{{ $slot }}` (composant Blade Breeze), ce qui est **incompatible** avec `@section`.
Le layout correct est `layouts/master.blade.php` qui utilise `@yield('content')`.

| `patients/create.blade.php` | `layouts.app` | `layouts.master` |
| `patients/show.blade.php` | `layouts.app` | `layouts.master` |

| `rendezvous/create.blade.php` | `layouts.app` | `layouts.master` |

### 2.4 ASSURANCES EN DUR - Donnees statiques au lieu de la BDD

**Fichier** : `app/Http/Controllers/PatientController.php`

Les methodes `create()` et `edit()` contenaient des assurances codees en dur :

```php
// AVANT (incorrect)
$assurances = collect([
    (object)['id' => 1, 'nom' => 'CNAMGS'],
    (object)['id' => 2, 'nom' => 'AXA GABON'],
    (object)['id' => 3, 'nom' => 'NSIA'],
    (object)['id' => 4, 'nom' => 'ASCOMA']
]);

// APRES (correct)
$assurances = Assurance::all();
```

**Impact** : Les assurances ajoutees via l'interface d'administration n'apparaissaient jamais dans le formulaire patient.

### 2.6 FILTRES PATIENTS NON FONCTIONNELS

**Fichier** : `app/Http/Controllers/PatientController.php`

```php
// AVANT - filtres ignores
public function index() {
    $patients = Patient::with('medecin')->latest()->paginate(10);
}

// APRES - filtres actifs
public function index(Request $request) {
    $patients = Patient::with('medecin')
        ->filter($request->only(['search', 'medecin_id', 'est_assure']))
        ->latest()
        ->paginate(10)
        ->withQueryString();
}
```

**Impact** : La recherche et les filtres de la vue `patients/index` ne produisaient aucun resultat filtre. `withQueryString()` ajoute pour conserver les filtres lors de la pagination.

## 4. Points d'attention restants

| # | Sujet | Severite | Detail |
|---|-------|----------|--------|
| 1 | Migration dupliquee `est_assure` | Faible | Deux migrations ajoutent la meme colonne. La seconde a un garde `hasColumn` donc pas de crash, mais c'est du bruit. |
| 2 | Migration mal nommee | Faible | `2026_03_25_124511_add_est_assure_to_patients_table.php` ajoute en realite `description` a `assurances`, pas `est_assure` a `patients`. |
| 3 | Migrations admin vides | Faible | `add_is_admin_to_users_table` et `add_admin_fields_to_users_table` sont vides (up/down ne font rien). |
| 4 | PHP absent du PATH | Info | `php artisan` n'est pas executable depuis le terminal actuel. Les routes n'ont pas pu etre validees par artisan. |
| 5 | CSRF sur toggle AJAX | Moyen | `toggleDispo` recoit des requetes POST via axios. S'assurer que le meta tag CSRF est bien lu par axios globalement. |

## 19. Securisation du deploiement - Niveau 1 (10 avril 2026)

Demande utilisateur : "mettre en place un systeme de securite pour proteger l'application
en ligne". Contexte : environnement de TEST (pas de vraies donnees patient), donc niveau
de securite proportionne (pas de 2FA ni packages composer). Aucune modification du code
applicatif - tout passe par les fichiers `.htaccess`.

### 19.1 Mesures ajoutees dans `public/.htaccess`

**Headers HTTP de securite (mod_headers)** :
- `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload` (HSTS 1 an)
- `X-Frame-Options: DENY` (anti-clickjacking)
- `X-Content-Type-Options: nosniff` (anti-MIME sniffing)
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: geolocation=(), microphone=(), camera=()...` (desactive APIs sensibles)
- `Cross-Origin-Opener-Policy: same-origin`
- `Cross-Origin-Resource-Policy: same-site`
- `X-Powered-By` et `Server` masques (`Header unset`)

**Blocage des fichiers sensibles** (`<FilesMatch>`) :
- `.env*`, `.git*`, `composer.json/lock`, `package*.json`, `webpack/vite/tailwind/postcss.config.js`
- `phpunit.xml*`, `artisan`, `server.php`, `.htpasswd`, `.user.ini`
- `README*`, `CHANGELOG*`, `LICENSE*`
- Extensions : `*.log`, `*.bak`, `*.backup`, `*.old`, `*.orig`, `*.swp`, `*.tmp`, `*.sql`,
  `*.sqlite*`, `*.yml`, `*.yaml`, `*.ini`, `*.conf`, `*.config`, `*.lock`, `*.md`

**Blocage des dossiers caches** (`RedirectMatch 404`) :
- `/.git`, `/.svn`, `/.hg`, `/.bzr`

**Blocage des methodes HTTP inutilisees** :
- `TRACE`, `TRACK`, `CONNECT` (anti cross-site tracing)

**Blocage des User-Agents de scanners malveillants** (regex case-insensitive) :
- nikto, sqlmap, masscan, nmap, fimap, nessus, whatweb, openvas, jbrofuzz, libwhisker,
  webshag, w3af, arachni, x-scanner, scanner, skipfish, ZmEu, Morfeus, zgrab, gobuster,
  dirbuster, wfuzz, hydra, Wprecon
- User-Agent vide aussi bloque

**Hardening serveur** :
- `Options -Indexes -MultiViews` (pas d'enumeration de dossiers)
- `php_flag expose_php Off` (masque la version PHP dans les headers)

### 19.2 Mesures ajoutees dans `deploy/htaccess-laravel-deny` (renforce 2026-04-10)

Fichier a placer dans `htdocs/laravel/.htaccess` (deja en place sur le serveur, mais
nouvelle version a re-uploader pour beneficier des protections supplementaires).
Strategie : defense en profondeur sur 8 niveaux. Si l'un tombe (config Apache, override,
faille), les autres tiennent.

| Niveau | Mesure | Protege contre |
|--------|--------|----------------|
| 1 | `Require all denied` (Apache 2.4+) | Acces HTTP direct a tout fichier |
| 2 | `Order deny,allow` + `Deny from all` (Apache 2.2 fallback) | Idem sur vieux Apache |
| 3 | `FilesMatch` bloquant `.php/.phtml/.phar/.php3-8/.pht/.phtm/.cgi/.pl/.py/.jsp/.asp/.aspx/.sh/.exe/.bat/.cmd` (case-insensitive) | Extensions executables meme si Niveau 1/2 tombe |
| 4 | `SetHandler None` + `ForceType text/plain` sur les memes extensions | Apache ne sait plus quoi executer, sert en texte brut |
| 5 | `RemoveHandler` / `RemoveType` / `AddType text/plain` via `mod_mime` | Desassociation complete du handler PHP/CGI |
| 6 | `php_admin_flag engine off` + `allow_url_fopen off` + `allow_url_include off` (mod_php/7/8) | Moteur PHP desactive ; `php_admin_flag` NON surchargeable par `ini_set()` contrairement a `php_flag` |
| 7 | `Options -Indexes -ExecCGI -MultiViews -Includes -FollowSymLinks` | Pas de listing, pas de CGI, pas de negociation, pas de SSI, pas de traversal symlink |
| 8 | `X-Content-Type-Options: nosniff` + `X-Frame-Options: DENY` + unset `X-Powered-By` | Fuite de type MIME, clickjacking, fingerprinting |

Changements cles par rapport a la version precedente :
- Extensions bloquees etendues (+ cgi/pl/py/jsp/asp/aspx/sh/exe/bat/cmd, + case-insensitive)
- Ajout de `SetHandler None` et `ForceType text/plain` (Niveau 4, nouveau)
- Ajout de `RemoveHandler`/`RemoveType`/`AddType` via `mod_mime` (Niveau 5, nouveau)
- `php_flag engine off` remplace par `php_admin_flag engine off` (non-surchargeable)
- Ajout `allow_url_fopen off` et `allow_url_include off` (anti-RFI)
- `Options` etendu avec `-Includes` (anti-SSI) et `-FollowSymLinks` (anti-traversal)
- Ajout de headers defensifs (Niveau 8, nouveau)

## 16. Script chmod 775 jetable pour deploiement mutualise (10 avril 2026)

Demande utilisateur : appliquer chmod 775 sur les dossiers Laravel inscriptibles par
le serveur web, en sachant que :
- Windows (NTFS) ne stocke pas les bits POSIX rwx, donc impossible en local
- L'hebergement page.gd est mutualise sans acces SSH
- Le workflow actuel utilise FTP pour uploader

| Element | Valeur |
|---------|--------|
| Fichier cree | `deploy/chmod.php` |
| Emplacement serveur | `htdocs/chmod.php` (a cote de `migrate.php`) |
| Cibles chmod | `htdocs/laravel/bootstrap/cache` et `htdocs/laravel/storage` (recursif) |
| Mode applique | `0775` |
| Token securite | genere via `openssl rand -hex 32` le 2026-04-10 |
| URL d'appel | `https://mnrdv.page.gd/chmod.php?token=<token>` |
| Cycle de vie | jetable - a SUPPRIMER du serveur immediatement apres execution |

- Token compare en strict (`===`), 403 sinon
- Parcours `RecursiveIteratorIterator` avec `SELF_FIRST` pour traiter le dossier racine
